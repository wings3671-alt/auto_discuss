## 1. 项目是什么

一个本机命令行"调度器" `auto_discuss.py`:让多个 AI(以及用户本人)围绕一份 Markdown 文档,**自动一轮一轮交替发表意见**,把一个议题讨论深、讨论透。

- **协作载体**:一份 Markdown 文件(如 `discuss.md`),每个参与者把自己那一轮追加进去。
- **参与者**通过本机 CLI 唤醒,**只走各自订阅,绝不走 API 计费**。
- **脚本只做"调度"** —— 判断轮到谁、唤醒谁、维护状态。它不生成讨论内容。

---

## 2. 当前现状(已完成,实测可用)

`auto_discuss.py` 目前是**【两方全自动版】**:Claude Code CLI 与 Codex CLI 交替发言,跑满 N 轮或双方达成共识即止。已实测跑通(用一场"门店门头灯光方案"讨论验证,2 轮收敛)。

核心机制(细节见代码):

- **状态块**:文档顶部一段 `<!-- AUTO_DISCUSS ... -->` HTML 注释,记 `round / turn / status / max_rounds / pending_final` 等,**由脚本独家维护**(参与者不许碰)。
- **轮流**:`turn` 在 `CLAUDE` / `CODEX` 间切换,每完成一对(Claude+Codex)`round` 加 1。
- **规则 B(双方共识才停)**:某方在自己那轮写独立一行 `FINAL_DONE` → 程序记 `pending_final`;对方下一轮也写 `FINAL_DONE` → 双方共识 → 停。单方写不停。
- **退出条件**:达 `max_rounds` / 状态块 `stop:true` / CLI 返回非 0 / 本轮无有效产出。
- **安全**:每轮带时间戳备份(`discuss.backup.YYYYMMDD-HHMMSS.md`)、文件锁(`discuss.lock`)、原子写、永不覆盖前文、检测到前文被篡改则自动从备份回滚。
- **计费**:调子进程前剔除 `*_API_KEY` 环境变量,确保走订阅而非 API。

**已验证可用的 CLI 调用方式**(新对话改 CLI 调用时以此为准):

- Claude:`claude -p --permission-mode acceptEdits "<prompt>"`,`cwd` 设为文档所在目录。
- Codex:`codex -a never exec -C <目录> --skip-git-repo-check -s workspace-write "<prompt>"`。

## 4. 【阶段 1a】详细规格 —— 新对话的开发任务

这一节是经过与用户逐条追问敲定的最终规格,可直接抽成 `1a规格.md`。

### 4.1 轮次结构
一轮 = **Claude → Codex → 用户**,然后进入下一轮。用户永远压轴。

### 4.2 运行方式
程序常驻、**一轮一停**。Claude、Codex 两棒程序自动连跑;到用户这一棒,程序停下、轮询等待用户。

### 4.3 用户回合机制
- 轮到用户时,程序往讨论文档**末尾追加一个占位段**:

```
### 🔴 用户 · 第 N 轮

──────────────  ✦  ──────────────
       老大,你的意见呢?
──────────────  ✦  ──────────────

› 在这行下面写你的意见……

› 写完另起一行:  over = 继续下一轮   ·   收工 = 结束讨论
```

- **占位段出现 = 通知用户**(用户在 VS Code 里盯着文档,看到它冒出来就知道轮到自己)。
- 用户在占位段下写意见,写完**另起一行**打 `over`(继续)或 `收工`(结束),保存文件。
- 程序每隔数秒**轮询**文档;检测到用户那一节里**独立整行**的 `over` 或 `收工`,即接棒。
- 等待用户期间,程序**不写讨论 Markdown**(所以用户此时编辑文档是安全的);唯一例外是刷新锁文件 mtime,见 4A.6。
- 等待**无限期**(用户不回就一直等)。轮询期间程序需定期 `touch` 锁文件,保持锁新鲜,免得被自身的过期判定误清。

### 4.4 用户的权威(重要)
- 用户那一轮 **= 决策者指令**,不是"第三个意见"。
- 下一轮 AI 的提示词**必须把用户上一轮的要求列为"本轮必须执行项"**,用命令级措辞(如:"用户=项目决策者,要求你做以下事,必须照办、不得忽略")。用户说"去查某网页",下一轮 AI 必须真去查(`claude -p` 自带 WebSearch/WebFetch,做得到)。
- 用户的发言**永远全文传给 AI,绝不压缩**(此约束延续到 1b)。
- 程序无法机械验证 AI 是否真照做;用户每轮在场,自行验收、不满意下轮点名。

### 4.5 结束规则
- **只有用户能终结讨论**:用户那轮写独立行 `收工` → 结束。
- AI 写 `FINAL_DONE` 在 1a 里**仅是"建议收尾"的文字**,程序不对它做任何动作(它只是给用户看的)。
- 程序真正的停止开关只有三个:用户写 `收工`、达轮数上限、报错。
- 轮数上限 = **12**(纯兜底,基本碰不到,因为用户每轮在场)。

### 4.6 标记约定(全部"独立整行"匹配)
| 标记 | 谁写 | 含义 |
|---|---|---|
| `over` | 用户 | 继续下一轮 |
| `收工` | 用户 | 结束讨论 |
| `FINAL_DONE` | AI | 建议收尾(1a 中仅文字,程序不响应) |

**"独立整行"是硬要求** —— 防止这些词出现在句子中间被误判(见第 7 节坑 #3)。

### 4.7 兼容旧模式
默认三方;加 `--no-user` 开关 → 退回旧的两方全自动行为(旧代码路径保留,不删)。

### 4.8 文件呈现(纯 Markdown)
- **不做** HTML 视图、不做文字颜色、不做提示音(用户已明确取消这些)。
- 每个参与者发言用**独立标题 + 分隔线**,保证一眼看清谁说的:
  - `### 🟡 Claude Code · 第 N 轮`
  - `### 🟢 Codex · 第 N 轮`
  - `### 🔴 用户 · 第 N 轮`
  - 各节之间用 `---` 分隔。
- emoji 圆点(🟡🟢🔴)是纯 Markdown 下的"伪颜色",任何 Markdown 预览都显示。

### 4.9 多讨论保留
- 一个讨论一个文件,程序靠 `--file` 指定。
- 约定:新讨论用新文件,统一放一个文件夹(如 `~/Desktop/讨论记录/门头灯_0517.md`)。旧讨论文件保留、永不覆盖。这只是使用习惯,不需做成功能。

### 4.10 沿用现状的全部安全机制
带时间戳备份、文件锁、原子写、永不覆盖前文、状态块脚本独家维护、防篡改自动回滚 —— 全部保留。

## 4A. 1a 状态机与解析规则(实现前必须钉死)

> 本节为采纳 Codex 评审后新增。这些细节不写死,实现时会靠猜 —— 必须严格照此实现。

### 4A.1 状态机:`turn` 为三值枚举
状态块的 `turn` 字段在 1a 中扩展为三值:**`CLAUDE` / `CODEX` / `USER`**。必须写进状态块、持久化到文件,**不得用临时变量绕过** —— 否则程序中途被杀、断点续跑时状态会乱。
轮转:`CLAUDE → CODEX → USER → CLAUDE`(下一轮);`round` 在 `USER` 完成后 +1。

### 4A.2 用户段的机器锚点
用户占位段除可见标题外,加一对**不可见 HTML 注释锚点**:

```
<!-- USER_TURN_START round=N -->
### 🔴 用户 · 第 N 轮
…(占位横幅 / 提示语 / 用户意见 / over|收工)…
<!-- USER_TURN_END round=N status=waiting -->
```

程序**只解析"最后一个 `USER_TURN_START` 之后"的内容**来找标记、提取意见。好处:① 旧轮里的 `over`/`收工` 永不会被误判;② 程序被杀后重启,能靠锚点重新定位"当前用户段"和轮次,断点续跑稳。用户回合结束后,程序把对应 `USER_TURN_END` 的 `status` 由 `waiting` 改为 `done`。

### 4A.3 标记解析规则(精确定义)
- 只在**最新用户段内**、按**「独立整行」**匹配 `over` / `收工`。
- 匹配前先 strip 该行首尾空白**及尾部标点** —— 所以 `over`、`Over`、`over.`、` over ` 都算数(**对非技术用户从宽,降低打错卡住的概率**)。
- `over` 大小写不敏感;`收工` 按中文原文匹配。
- 同一用户段内若 `over` 和 `收工` 同时出现:以**最后出现的那个**为准(代表用户最终意图)。
- 用户段已有实质内容、但找不到任何有效标记:程序**继续等待、不推进**,并在终端/日志打印一行提示"未检测到 over / 收工"。

> 注:此处对"大小写/标点变体"的处理比 Codex 建议的"严格不认变体"更宽容 —— 这是刻意的,因为用户非程序员,严格匹配会因一个标点之差静默卡住、体验差。

### 4A.4 用户意见的提取算法
传给下一轮 AI 的"用户上一轮发言" = 最新用户段中,**从占位提示语之后、到 `over`/`收工` 标记之前**的全部文本,原样注入下一轮 prompt。
**不要**把占位横幅(「老大,你的意见呢?」)、提示语(「写完另起一行…」)、控制标记一并传给 AI —— 那是噪音。

### 4A.5 发言标题格式统一(+ 校验正则同步)
1a 起,三方发言标题**统一**为:
`### 🟡 Claude Code · 第 N 轮` / `### 🟢 Codex · 第 N 轮` / `### 🔴 用户 · 第 N 轮`(emoji + 中文间隔点 `·`)。
⚠️ **实现陷阱**:现有代码的 `round_heading_re` 匹配的是**旧格式** `### Claude Code - 第 N 轮`。改标题格式时,**校验正则必须同步改成新格式**,否则会出现"AI 写了新标题、校验仍按旧格式找 → 误判本轮没产出 → 报错"。改一处必改另一处。

### 4A.6 等待用户期间的写操作界定
"等待期间只读不写"的**精确含义**:
- **不得写讨论 Markdown 文件**(讨论内容、占位段在进入等待前就已写好)。
- **允许且仅允许**定期刷新 `discuss.lock` 的 mtime(`touch`),使锁不被自身的过期判定误清。
- 状态块也在进入等待前更新好,等待期间不动。

## 5. 跨阶段要求:讨论守则(批判性思维)

用户明确要求:讨论中各方**不要互相附和**,要拿出**批判性思维 + 发散性思维**,让讨论有深度、有广度。

落地方式:

- 把这份"守则"**每一轮都注入提示词** —— 这是唯一可靠渠道(AI 每轮是无记忆的新进程,只写在文档总纲里不保证被当回事)。
- 同时在配置(脚本常量,或一个 `讨论守则.md` 文件)里留一份**可编辑的守则文本**,用户可随时改措辞。
- 守则大意:"你必须对其他参与者的观点做批判性审视,指出漏洞与风险,提出发散的替代思路,不要无条件附和。"

## 6. 【阶段 1b】上下文优化(待 1a 验收后做,方向已定)

> 以下是用户提供的设计,加上 Claude 评审后的**关键修正**。1b 开工前还需就"指令格式"单独定一轮。

**要解决的问题**:现在每轮把讨论 md **全文**读入。单轮不撑爆窗口,但累计代价是平方级(每轮重读全文,N 轮累计 token ≈ 三角形面积)。三个代价:成本、延迟、长讨论里注意力被稀释导致后期发言变水。

**目标**:每轮喂进上下文的量是个小常量,不随轮数增长。

**设计**:每轮上下文 = 对方上一轮新增发言 + 结构化滚动状态 +(需要时)按需回查的原文片段。

1. **结构化滚动状态(核心)**:独立文件 `STATE.md`,四个固定栏目 ——
   ① 已达成的结论 ② 待解决的问题 ③ 已否决的方案+原因 ④ 关键约束/前提。
   原则:这是一份**被增改的清单**,不是被重写的文章。推理过程可压掉,但结论/待办/否决理由/约束等硬信息原样保留。
2. **每轮流程**:程序喂 `STATE.md` 全文 + 对方最近一轮发言(不再喂全文)→ AI 输出"讨论回复"(追加到讨论 md)+ "状态更新指令" → 程序按指令更新 `STATE.md` → 触发下一轮。
3. **回查兜底**:讨论 md 每轮用稳定锚点分段;`STATE.md` 条目带指针(如"A 方案见第 7 轮");AI 发现状态不够用时可直接读讨论 md 任意历史轮次。原文始终完整在磁盘,信息从不丢。

**Claude 的关键修正(必须在 1b 设计阶段先钉死)**:

> "状态更新指令"**绝不能是自由中文文本**让程序去解析 —— 那等于把设计想消除的"传话漂移"换个地方重新引入。
> 必须定义**严格的机器可读格式**。**采纳 Codex 评审:用 JSON,不用自定义中文命令** —— `ADD 待解决: …` 这类自定义语法仍需自己写解析器;改用 JSON 则有标准解析器(`json.loads`)和 schema 校验。AI 在回复里输出一个固定标记的代码块(如 ` ```STATE_OPS ` … ` ``` `),内容是合法 JSON,例如:
>
> ```json
> {"ops":[{"op":"add","section":"open_questions","text":"…","source":"round-3-codex"}]}
> ```
>
> 程序只接受符合预定 schema 的 JSON,解析失败 / 不合 schema 就要求该 AI 重试。
> 建议**单一写入点**:AI 只写讨论 md(讨论回复 + 末尾的 STATE_OPS 代码块都在里面),程序从那里提取再更新 `STATE.md`。比"AI 同时写两个文件"可靠。
> `STATE_OPS` 的完整 JSON schema,在 1b 开工前要单独定一轮。

## 7. 已踩过的坑(务必避免重蹈!)

开发 1a 若改动 CLI 调用相关代码,以下是血泪教训:

1. **嵌套环境变量污染**:从一个 Claude Code 会话里启动本脚本时,子进程继承了 `CLAUDE*` / `AI_AGENT` 等环境变量,被调起的 `claude` 会误以为自己嵌套在另一个会话里,**空跑、1 秒秒退、什么都不做**。→ 调 `claude` 前必须剔除所有以 `CLAUDE` 开头的变量及 `AI_AGENT`。
2. **`--add-dir` 是变长参数**:`claude --add-dir <目录> "<prompt>"` 会把后面的 prompt 也吞成"另一个目录",导致 claude 报"收不到 prompt"。→ **不要用 `--add-dir`**;靠 `cwd`(工作目录设为文档所在目录)给文件访问权。
3. **`FINAL_DONE` 全文扫描误判**:文档别处(需求说明、引用、本文档)出现 `FINAL_DONE` 字样,会让程序一启动就误判讨论结束。→ 只在"本轮新追加区域"里、按**独立整行**正则检测。同理本轮标题校验也只在新增区域做。
4. **状态块识别区域**:文档正文里可能把 `AUTO_DISCUSS` 状态块作为示例引用。→ 只在**第一个 `## ` 标题之前**的区域识别真状态块。
5. **codex 审批**:非交互无人值守必须 `codex -a never exec …`,否则可能卡在审批等待。
