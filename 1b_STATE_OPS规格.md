# 阶段 1b · STATE_OPS 指令格式规格

> 本文件是交接文档第 6 节要求的"1b 开工前单独定的一轮"的产出,经与用户
> 逐点敲定(2026-05-17)。1b 实现以本文件为准。
> **本规格只管 `STATE_OPS` 指令格式 + `STATE.md` 结构**;其余 1b 实现细节
> (喂什么上下文、讨论 md 回查锚点等)见末尾"留给 1b 实现阶段"一节。

---

## 1. 这是干什么的

1b 要解决"每轮把讨论 md 全文读入 → 累计 token 平方级增长"。办法:不再喂
全文,改喂一份**滚动状态** `STATE.md`。

- `STATE.md` = 一块**滚动白板**,贴着四类便签。
- `STATE_OPS` = AI 每轮发完言后交给程序的一张**「便签更新单」**(贴/改/撕)。
- **程序是 `STATE.md` 的唯一执笔人**(同 1a 状态块铁律:防"传话漂移")。
- AI 永远不直接编辑 `STATE.md`,只输出 `STATE_OPS`。

---

## 2. STATE.md 的结构

每个讨论一份独立状态文件,放在讨论 md 旁边,命名 `<讨论文件名>.state.md`
(例:`门头灯_0517.md` → `门头灯_0517.state.md`)。讨论文件不存在状态文件
时,程序自动创建一份四栏目皆空的初始 `STATE.md`。

四个**固定栏目**(交接文档第 6 节):

| 栏目 | 机器 key | 便签编号前缀 |
|---|---|---|
| 已达成的结论 | `conclusions` | `C` |
| 待解决的问题 | `open_questions` | `Q` |
| 已否决的方案(含原因) | `rejected` | `R` |
| 关键约束 / 前提 | `constraints` | `K` |

形态(人类可读 Markdown,栏目顺序固定):

```
<!-- STATE_META next_C=3 next_Q=5 next_R=2 next_K=3 -->
# 讨论滚动状态

## 已达成的结论
- [C1] 推送渠道用 Telegram,不用邮件   <!-- touched: r2-codex -->
- [C2] 每天推送 2 次:晨间和傍晚       <!-- touched: r3-claude -->

## 待解决的问题
- [Q2] 潮汐数据用哪个 API,免费额度够不够   <!-- touched: r2-claude -->
- [Q3] 推送时间固定钟点还是按潮汐动态算     <!-- touched: r4-codex -->

## 已否决的方案
- [R1] 做成独立 App —— 否决:开发成本高,用户只需被动接收   <!-- touched: r1-claude -->

## 关键约束 / 前提
- [K1] 用户不写代码,最终要双击运行   <!-- touched: r1-claude -->
- [K2] 只走本机订阅,不付费调 API     <!-- touched: r2-codex -->
```

### 2.1 便签编号(ID)
- 格式正则:`^[CQRK][0-9]+$`。前缀决定它属于哪个栏目。
- **由程序分配**,AI 不许自己造编号。
- **单调递增、永不复用**:撕掉 `Q3` 后,新问题拿 `Q4`,`Q3` 这个号永久作废
  —— 防止 AI 引用到一个被回收的旧号、改错便签。
- 下一个可用号记在文件顶部 `STATE_META` 注释里,由程序维护。

### 2.2 来源指针 `touched`
- 每条便签行尾一个 HTML 注释 `<!-- touched: r<轮次>-<说话者> -->`。
- 由**程序自动盖章**:程序知道当前第几轮、这张更新单是谁交的(claude/codex)。
  AI 不用、也不许自己填。
- 用途:回查兜底 —— AI 觉得某条便签信息不够时,照指针去翻讨论 md 对应轮次。
- `add` / `update` 都把 `touched` 刷成当前轮次;`remove` 整行删掉。

### 2.3 谁能改 STATE.md
- **只有程序**,且只通过应用 `STATE_OPS` 来改。
- 用户**不直接编辑** `STATE.md`。用户若发现状态写错了,在自己那一轮发言里
  说出来(如"结论 C2 不对,应改成 X"),由下一轮 AI 用 `update` 指令修正。
  这样维持"单一写入者",可靠性同 1a 状态块。

## 3. STATE_OPS 指令块

### 3.1 位置与形态(方案乙:程序执笔,2026-05-17 用户拍板)
- AI **不编辑任何文件**。它每轮把"发言正文 + 末尾的 STATE_OPS 代码块"作为
  **回复文本**输出(走 CLI 的 stdout)。
- **程序执笔**:程序收下这段回复,替 AI 把它写进讨论 md(加好小节标题与
  `<!-- TURN -->` 锚点),再从中提取 STATE_OPS 块、据此更新 `STATE.md`。
- STATE_OPS 块是一个**带信息串 `STATE_OPS` 的围栏代码块**(```STATE_OPS … ```),
  写在发言正文之后。它**留在讨论 md 里可见**(用户决策):用户每轮能看到
  各 AI 对白板做了什么、可当场抓错。
- 程序在 AI 本轮回复里找这个块:**必须恰好有一个**;0 个或 ≥2 个 → 不合格。
- 好处:AI 上下文里只有程序喂的那一小份(省 token 目标稳稳达成);AI 碰不到
  文件 → "AI 改坏前文"的风险消失;程序成为两个文件唯一的笔。

### 3.2 每轮必交
- AI **每个回合都必须**输出 STATE_OPS 块。本轮无需改白板时,交空单:
  `{"ops":[]}`。
- 强制每轮交单是为了防 AI 偷懒不维护白板,导致整个 1b 机制慢慢烂掉。
- 用户回合**不产生** STATE_OPS(用户不写 JSON);用户的要求由下一轮 AI
  消化后,用它自己的 STATE_OPS 反映进白板。

### 3.3 JSON schema

块内容是一个 JSON 对象:

```
{ "ops": [ <操作>, <操作>, ... ] }      // ops:数组,必填,可为空 []
```

`<操作>` 是且仅是以下三种之一,字段必须**严格匹配**(多出未知字段即判不合格):

```
add     {"op":"add",    "section":<栏目key>, "text":<非空字符串>}
update  {"op":"update", "id":<已存在编号>,    "text":<非空字符串>}
remove  {"op":"remove", "id":<已存在编号>}
```

- `section` ∈ `conclusions` / `open_questions` / `rejected` / `constraints`。
- `id` 必须匹配 `^[CQRK][0-9]+$`,且当前在 `STATE.md` 里**真实存在**(live)。
- `text` 不可为空白。
- `add`:在对应栏目**末尾追加**一条新便签,程序分配新编号。
- `update`:把该编号便签的 `text` **整体替换**(不是打补丁);栏目不变
  (编号前缀已锁定栏目)。
- `remove`:删掉该编号便签整行。
- "某问题被解决" = 同一张单里写 `remove` 旧 Qx + `add` 一条新结论。不设
  独立的 move/resolve 操作 —— 3 种操作已够用,越少越不易错。
- `rejected` 栏目的便签:`text` 里**必须同时含方案与否决原因**(由提示词约束,
  schema 层面 text 对四个栏目一视同仁,不拆子字段)。

### 3.4 校验与应用
- **原子**:程序先校验整张单的**每一条**;任何一条不合法 → **整张单退回、
  一条都不应用**,要求该 AI 重新输出。绝不半截应用。
- 不合法包括:JSON 解析失败、`ops` 不是数组、未知 `op`、缺必填字段、出现
  未知字段、`section` 非四选一、`id` 格式错或指向不存在的便签、`text` 空白、
  同一编号在一批里被引用多次。
- **重试**:不合格 → 把错误原因告诉该 AI、要它重答,重试 1 次。
- **软兜底**(实现时定,2026-05-17):重试后仍不合格,**不**中断讨论 ——
  本轮跳过白板更新(等同空单 `{"ops":[]}`),并在讨论 md 留一行可见警告,
  讨论照常继续。理由:为一次 JSON 失误就掐掉用户投入的整场讨论代价太重;
  用户每轮在场,看到警告自会处置。

## 4. 一个完整例子

**白板原样**(讨论到第 4 轮后,`STATE_META next_C=3 next_Q=4 …`):

```
## 待解决的问题
- [Q3] 推送时间固定钟点还是按潮汐动态算   <!-- touched: r4-codex -->
## 已达成的结论
- [C2] 每天推送 2 次:晨间和傍晚         <!-- touched: r3-claude -->
```

**第 5 轮 Codex 发言末尾交的更新单:**

````
```STATE_OPS
{"ops":[
  {"op":"remove","id":"Q3"},
  {"op":"add","section":"conclusions","text":"推送时间按当天潮汐动态计算,不用固定钟点"},
  {"op":"add","section":"open_questions","text":"动态时间怎么避免在深夜推送打扰用户"}
]}
```
````

**程序应用后:** Q3 行删除;新增 `[C3]`(动态计算…,touched r5-codex);
新增 `[Q4]`(深夜打扰…,touched r5-codex);`STATE_META` 更新为
`next_C=4 next_Q=5 …`。全程按编号精确操作,不靠猜"AI 想改哪条"。

## 5. 留给 1b 实现阶段(本规格不含,实现时再定/照交接文档)

- **喂给 AI 的上下文**:每轮 = `STATE.md` 全文 + 对方最近一轮发言 + 讨论守则
  +(若上一棒是用户)用户发言全文。不再喂讨论 md 全文。细节按交接文档第 6 节,
  实现时确认。
- **讨论 md 回查锚点**:讨论 md 每轮加稳定锚点分段,供 AI 按 `touched` 指针
  回查历史轮次原文。原文始终完整在磁盘。
- 提示词如何讲清"什么算结论/问题、怎么合并精简便签防止白板膨胀"。
- 与 1a 的衔接:三方结构、用户权威、全部安全机制(备份/锁/原子写/防篡改)不变。
