# 阶段 1a 规格 · 三方轮流(Claude + Codex + 用户)

> 本文件抽自《设计方案与交接文档.md》第 4 节、第 4A 节,作为 1a 开发的权威规格存档。
> 抽取日期:2026-05-17。开发以本文件为准;与交接文档冲突时,以交接文档为准。

---

## 1. 轮次结构
一轮 = **Claude → Codex → 用户**,然后进入下一轮。用户永远压轴。

## 2. 运行方式
程序常驻、**一轮一停**。Claude、Codex 两棒程序自动连跑;到用户这一棒,程序停下、轮询等待用户。

## 3. 用户回合机制
- 轮到用户时,程序往讨论文档**末尾追加一个占位段**:

```
### 🔴 用户 · 第 N 轮

──────────────  ✦  ──────────────
       老大,你的意见呢?
──────────────  ✦  ──────────────

› 在这行下面写你的意见……

› 写完另起一行:  over = 继续下一轮   ·   收工 = 结束讨论
```

- **占位段出现 = 通知用户**(用户在 VS Code 里盯着文档,看到它冒出来就知道轮到自己)。
- 用户在占位段下写意见,写完**另起一行**打 `over`(继续)或 `收工`(结束),保存文件。
- 程序每隔数秒**轮询**文档;检测到用户那一节里**独立整行**的 `over` 或 `收工`,即接棒。
- 等待用户期间,程序**不写讨论 Markdown**;唯一例外是刷新锁文件 mtime(见第 13 节)。
- 等待**无限期**(用户不回就一直等)。轮询期间程序需定期 `touch` 锁文件,保持锁新鲜。

## 4. 用户的权威
- 用户那一轮 **= 决策者指令**,不是"第三个意见"。
- 下一轮 AI 的提示词**必须把用户上一轮的要求列为"本轮必须执行项"**,用命令级措辞
  (如:"用户=项目决策者,要求你做以下事,必须照办、不得忽略")。
- 用户的发言**永远全文传给 AI,绝不压缩**(此约束延续到 1b)。
- 程序无法机械验证 AI 是否真照做;用户每轮在场,自行验收。

## 5. 结束规则
- **只有用户能终结讨论**:用户那轮写独立行 `收工` → 结束。
- AI 写 `FINAL_DONE` 在 1a 里**仅是"建议收尾"的文字**,程序不对它做任何动作。
- 程序真正的停止开关只有三个:用户写 `收工`、达轮数上限、报错。
- 轮数上限 = **12**(纯兜底)。

## 6. 标记约定(全部"独立整行"匹配)
| 标记 | 谁写 | 含义 |
|---|---|---|
| `over` | 用户 | 继续下一轮 |
| `收工` | 用户 | 结束讨论 |
| `FINAL_DONE` | AI | 建议收尾(1a 中仅文字,程序不响应) |

## 7. 兼容旧模式
默认三方;加 `--no-user` 开关 → 退回旧的两方全自动行为(旧代码路径保留,不删)。

## 8. 文件呈现(纯 Markdown)
- **不做** HTML 视图、不做文字颜色、不做提示音。
- 每个参与者发言用独立标题 + 分隔线:
  - `### 🟡 Claude Code · 第 N 轮`
  - `### 🟢 Codex · 第 N 轮`
  - `### 🔴 用户 · 第 N 轮`
  - 各节之间用 `---` 分隔。

## 9. 多讨论保留
一个讨论一个文件,程序靠 `--file` 指定。新讨论用新文件,旧文件永不覆盖。

## 10. 沿用现状的全部安全机制
带时间戳备份、文件锁、原子写、永不覆盖前文、状态块脚本独家维护、防篡改自动回滚。

## 4A. 状态机与解析规则(实现前必须钉死)

### 11. 状态机:`turn` 为三值枚举
- `turn` 扩展为三值:**`CLAUDE` / `CODEX` / `USER`**,必须写进状态块、持久化到文件,
  不得用临时变量绕过。
- 轮转:`CLAUDE → CODEX → USER → CLAUDE`;`round` 在 `USER` 完成后 +1。

### 12. 用户段的机器锚点
用户占位段除可见标题外,加一对不可见 HTML 注释锚点:

```
<!-- USER_TURN_START round=N -->
### 🔴 用户 · 第 N 轮
…(占位横幅 / 提示语 / 用户意见 / over|收工)…
<!-- USER_TURN_END round=N status=waiting -->
```

- 程序**只解析"最后一个 `USER_TURN_START` 之后"的内容**来找标记、提取意见。
- 用户回合结束后,程序把对应 `USER_TURN_END` 的 `status` 由 `waiting` 改为 `done`。

### 13. 标记解析规则(精确定义)
- 只在**最新用户段内**、按**「独立整行」**匹配 `over` / `收工`。
- 匹配前先 strip 该行首尾空白**及尾部标点** —— `over`、`Over`、`over.`、` over ` 都算数。
- `over` 大小写不敏感;`收工` 按中文原文匹配。
- 同段内 `over` 和 `收工` 同时出现:以**最后出现的那个**为准。
- 用户段已有实质内容但找不到有效标记:程序**继续等待、不推进**,终端打印一行
  "未检测到 over / 收工"。

### 14. 用户意见的提取算法
传给下一轮 AI 的"用户上一轮发言" = 最新用户段中,**从占位提示语之后、到
`over`/`收工` 标记之前**的全部文本,原样注入下一轮 prompt。
不把占位横幅、提示语、控制标记传给 AI。

### 15. 发言标题格式统一(+ 校验正则同步)
- 三方标题统一为:`### 🟡 Claude Code · 第 N 轮` / `### 🟢 Codex · 第 N 轮` /
  `### 🔴 用户 · 第 N 轮`(emoji + 中文间隔点 `·`)。
- ⚠️ **实现陷阱**:现有 `round_heading_re` 匹配旧格式 `### Claude Code - 第 N 轮`。
  改标题格式时,**校验正则必须同步改成新格式**。改一处必改另一处。

### 16. 等待用户期间的写操作界定
- **不得写讨论 Markdown 文件**。
- **允许且仅允许**定期刷新 `discuss.lock` 的 mtime(`touch`)。
- 状态块在进入等待前更新好,等待期间不动。
